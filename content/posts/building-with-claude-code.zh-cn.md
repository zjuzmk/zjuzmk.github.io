> 我想要一个双语、易维护、不锁平台的个人学术主页。WordPress 太重，Notion 太锁，手写 HTML 又不像研究者该做的事。

<!--more-->

## 为什么动手

我意识到一个研究者的网络门面很重要。Google Scholar 的 profile 太简陋，ResearchGate 的界面又太乱。我想有一个地方，能放我的研究项目、发表记录、田野照片，以及偶尔的想法——中英文都要。我拖延了半年，因为所有选项都像妥协：WordPress 要维护插件，Notion 的内容锁在平台里，自己写代码又浪费精力。

## 技术方案

第一次尝试用的是 **Material for MkDocs**。优点很明显：内容纯 Markdown，天然双语插件，本地 dev server 能实时预览。我用了一个周末搭出了骨架——主页、发表、关于、五个研究项目页、还有一个带 Leaflet.js 的地图。

但问题很快来了：

- MkDocs 的 blog 插件和 i18n 插件不兼容，列表页渲染空白
- 自定义 CSS 被主题层层覆盖，改一个颜色要查三处
- 移动端导航栏在深色模式下对比度不够

几天后，我迁移到了 **Hugo + Hextra 主题**。原因很实际：Hugo 的模板系统更灵活，我还发现 Hextra 的设计更现代，而且构建速度比 MkDocs 快一个数量级。

迁移过程出奇地简单——内容文件直接从 `docs/` 搬到 `content/`，配置从 `mkdocs.yml` 转成 `hugo.yaml`，大部分 Markdown 不需要改动。唯一的折腾是重新调了一遍 CSS 配色。

## 协作流程

整个开发是在 [Claude Code](https://claude.ai/code) 里完成的，用中文交流。我的习惯是：

1. 用中文描述我想要什么（比如"首页要压缩成一张名片"）
2. 它提出两到三个方案，附带利弊
3. 我选一个，或者继续调整
4. 它写代码，我旁边看着
5. 本地预览，不满意就改

这种节奏适合我，因为我不需要切换语言思考。技术细节的英文术语它自然能处理，但"这个按钮太靠右了"这种主观判断用中文表达更准。

## 为什么是 Posts 不是 Blog

Blog 这个词自带预期——定期更新、主题统一、篇幅完整。我不想给自己定这些规矩。有时候只想贴一张田野调查时拍的烂图，配一句"这棵树长得真嚣张"；有时候读完半本书想记两句碎片。这些都不像 blog，更像 post。

这个栏目以后会塞满读书笔记、田野速记、城市观察照片、工具吐槽，以及没成型的想法。没有承诺，没有预期，没有 KPI。
