# 我和 Claude Code 搭了个网站

*2026年5月29日 · 标签: `ClaudeCode` `个人网站` `学术主页`*

---

想搭一个个人学术主页这件事，我想了大半年了。

和很多研究者一样，我想要的是：干净、中英双语、好维护。但每次一想到要动手就开始拖延——WordPress 太重，Notion 太锁，自己手写 HTML 又觉得是另一个周末的坑。

直到我试了试 [Claude Code](https://claude.ai/code)——就是你现在看到的这个工具。

## 开始

我往一个空文件夹里丢了份简历（Markdown 格式），然后问它："我们做个什么项目？"

它读了我的简历，知道我是做城乡研究的，然后给了四个建议。我选了 **搭个人学术网站**——实用、看得见成果、又是测试这个协作流程的好项目。

## 技术方案

我们选了 **Material for MkDocs**，理由很简单：

- **内容用 Markdown** — 我本来就大量用 `.md` 写东西，不需要学新的
- **天然双语** — 英文和中文，一键切换。不是"以后再加"，而是一开始就有
- **不锁平台** — 所有内容就是一堆 `.md` 文件，没有数据库，没有专有格式。将来想换工具，内容直接搬走
- **免费部署** — GitHub Pages，零成本上线

## 协作流程

整个开发过程就是不断重复这个循环：
1. 我用中文说想要什么
2. Claude 给出方案和利弊分析
3. 我拍板或调整方向
4. Claude 写代码，我看着
5. 打开 `localhost:8000` 实时看效果
6. 我反馈，再改

从"空文件夹"到"一个能看的、双语的、带配色方案的网站"，只用了几个小时。

## 现在建了什么

截至目前，这个网站已经有了：

- **主页** — 中英双语，干净的研究者简介和研究方向
- **发表记录** — 6 篇期刊论文 + 2 个著作章节 + 在撰手稿，全部可点击 DOI
- **研究项目** — 5 个独立项目页面，涵盖博士论文（崇礼延伸城市化）、北京冬奥会文化遗产、现场音乐与平台经济、单位制空间转型、大运河遗产保护
- **Blog** — 就是这个页面。没有用 Material 的内置 blog 插件（它和中英双语插件不兼容，折腾了好一阵子），改用手写的 markdown 页面，简单可控
- **交互地图** — Leaflet.js + OpenStreetMap，标注了崇礼、北京、杭州、深圳四个田野调查点，点击有说明
- **暗色模式** — 一键切换，晚上看不刺眼

## 几个让我意外的点

- **它真的读懂了上下文。** 我不需要反复解释"我是做什么的"。它读了我的简历，理解我的研究领域，能给出有针对性的建议。
- **不用反复跳出去搜教程。** 不用 Google "mkdocs i18n 怎么配置"——Claude 要么知道，要么现场读插件的源码。
- **全过程在 Git 里。** 所有内容、配置、设计决策都写在了 `SITE-PLAN.md` 里。项目是完整可迁移的。
- **它还会记笔记。** 我的偏好、项目配置、设计选择都存到了 memory 里，下次对话它还记得。
- **它能读论文。** 我把已发表的几篇 PDF 扔进文件夹，Claude 读完后自动提取了每个项目的研究问题、方法和核心发现，写了项目页面，我把关修改就行。

## Bug 也是学习过程

中间踩了不少坑：
- Blog 插件和 i18n 插件不兼容——列表页一片空白。在 `mkdocs serve` 重启了好几次才发现问题。最后放弃插件，用手写的 markdown 页面反而更干净
- 中英切换有个 404 bug——英文版的 alternate link 配成了 `/en/`，但实际上英文是默认语言，应该指向 `/`。改一行就修好了
- 从中文主页翻译过去的英文页面，一些专用词翻译不准——比如把"研究组"翻成了"Laboratory"，"建筑与城市科学"翻成了"Architecture and Urban Sciences"。后来我把英文 CV 放进项目文件夹，Claude 对照着全部修正了

## 小结

完美吗？不——有沟通不对位的时候，有 404 bug，有调了好几轮的 CSS，有插件兼容性搞了一小时的经历。但对于第一次用这种工具来说，这个节奏和质量是超出预期的。

如果你也是研究者，想要一个有模有样的网络门面但不想变成业余前端开发——**这个工作流值得试试。**
